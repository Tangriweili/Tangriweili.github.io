RuoYi 3.8.2（前后端分离）渗透测试报告

说明：本文档以项目测试与分析为目的，聚焦前端交互与与前后端分离后暴露的接口风险。文中留有截图占位，便于后续插图与证据补充。语言力求专业、通俗、结构清晰，减少逻辑性衔接词，便于直接作为测试交付物使用。

1 报告信息

  • 报告标题：RuoYi 3.8.2 前后端分离 渗透测试报告
  • 目标系统：若依（RuoYi-Vue）前端 + 后端 API
  • 版本：RuoYi-Vue 3.8.2
  • 数据库：ruoyi
  • 测试范围：主要聚焦前台（用户面）交互、前端逻辑、与后端 API 的授权/认证、输入校验与敏感信息暴露
  • 报告作者：测试团队
  • 测试时间：YYYY-MM-DD（请填写实际测试日期）

2 执行摘要（简洁）

  • 目标系统存在多类前端相关风险。部分风险可通过前端逻辑与不安全的 API 配合实现数据泄露或权限越界。部分漏洞为已知组件/版本问题，应优先修复。
  • 建议按风险等级优先处理：高 -> 中 -> 低。高危项应在修复前采取临时缓解措施（如关闭相关接口、增强网关校验）。

3 范围与限制

  • 范围仅限前台交互与前后端接口（/api/**、常见 public 路径）。
  • 未包含对后端物理主机、内网横向移动或底层操作系统级别的渗透。
  • 渗透过程中仅执行被授权测试动作。若需复现高危利用，请在隔离环境完成。

4 测试环境

  • 前端：RuoYi-Vue 3.8.2（构建产物或本地 dev）
  • 后端：Spring Boot API（默认上下游端点）
  • 数据库：MySQL，库名 ruoyi
  • 工具列表：Burp Suite、OWASP ZAP、curl、httpie、sqlmap（按需）、nmap、jq、浏览器控制台、F12 网络面板

5 测试方法与步骤概述

  1. 识别前端路由与静态资源，记录关键页面（登录、注册、找回密码、个人中心、文件下载上传、富文本编辑区）。
  2. 通过浏览器代理抓包，枚举前端发出的 API 请求，映射接口与参数。
  3. 对接口进行未授权访问、水平/垂直越权测试、参数篡改测试、IDOR 测试。
  4. 对可能存在的输入点进行 XSS、CSRF 与上传相关测试。
  5. 对文件类接口测试任意文件下载/读取、任意文件上传与后续文件访问。
  6. 分析前端使用的第三方库（如 jsencrypt）与本地存储策略，检查敏感信息是否保存在 cookie/localStorage 中。

6 关键发现（按风险等级分组）

高危（需立即处理）

A. 敏感信息以明文或可逆方式保存在 Cookie（已知组件风险）

  • 描述：若依前端在密码处理/登录流程中，使用了 jsencrypt.js 或自定义加密组件。部分版本存在将敏感信息以可恢复或明文方式写入 cookie 的问题。攻击者可通过 XSS 或同域脚本读取 cookie，从而窃取凭证或会话信息。
  • 影响：账户被劫持、数据泄露、横向越权。影响版本：RuoYi-Vue <= 3.8.9（3.8.2 属于受影响范围）。
  • 复现步骤（示例）：
    1. 在登录时观察 cookie 内容（浏览器开发者工具 -> Application -> Cookies）。
    2. 使用代理抓包，确认登录请求/响应中是否包含可识别的敏感字段。
    3. 模拟 XSS（或借助浏览器控制台）读取 document.cookie，确认是否能获取敏感条目。
  • 证据/截图占位：
    • 截图1：登录后 cookie 列表（包含敏感字段位置）。
    • 截图2：抓包显示 login 请求/response。
  • 建议修复：
    • 不在 cookie 中存储明文敏感信息。仅存储 HttpOnly, Secure 的 session id 或短期令牌。
    • 使用服务端完成敏感信息加密与验证，前端仅负责展示。
    • 若使用加密库，确保密钥不在前端暴露。

B. 任意文件下载 / 任意文件读取（历史高危类型，需核实）

  • 描述：若依早期某些接口（如 /common/download/resource）对输入路径未做严格限制，可能导致任意文件下载或读取。前后端分离后若保留类似通用下载接口，仍可能被滥用。
  • 影响：配置文件、数据库备份、敏感凭据泄露。可能导致进一步 RCE。
  • 复现步骤（示例）：
    1. 访问疑似下载接口：GET /common/download/resource?name=/etc/passwd 或 ?name=../../../../application.yml。
    2. 检查响应是否返回服务器文件内容。
  • 证据/截图占位：
    • 截图3：任意文件读取成功响应示例。
  • 建议修复：
    • 限制下载路径白名单。
    • 对路径参数做严格规范化与校验，禁止 .. 等相对路径。
    • 将文件存储操作限制在不可直接遍历的存储目录，使用映射 ID 而非原始文件路径。

中危（优先处理）

C. 前端可触发的反射型/存储型 XSS

  • 描述：输入点（如用户资料、富文本编辑、留言）若未对输入或输出做严格过滤与转义，可能被注入脚本。前端富文本处理时若信任 HTML，将扩大发现风险。
  • 影响：会话窃取、钓鱼、凭证泄露。对使用 cookie 存储敏感信息的系统影响更大。
  • 复现步骤（示例）：
    1. 在可编辑输入区域提交 <script>alert(1)</script> 或 payload。
    2. 刷新或在其他用户页面观察是否触发。
  • 证据/截图占位：
    • 截图4：注入点与触发弹窗的页面。
  • 建议修复：
    • 使用服务端与前端双重输出编码。
    • 对富文本输入采用安全白名单（允许的标签与属性）并使用库（如 DOMPurify）进行清洗。

D. CSRF 风险与不安全的 CORS 配置

  • 描述：若后端对跨域请求未做限制，或 CORS 允许 * 或任意来源，攻击者可在用户登录状态下发起跨站请求。部分 API 若只依赖 cookie 认证，将受到 CSRF 影响。
  • 影响：在已登录用户上下文下执行未授权操作。
  • 复现步骤（示例）：
    1. 构造跨站请求页面，自动提交对目标站点的 POST 请求。
    2. 若后端接受并执行，则存在 CSRF。
  • 证据/截图占位：
    • 截图5：响应头 Access-Control-Allow-Origin 与示例请求/响应。
  • 建议修复：
    • 对重要写操作开启 CSRF Token 校验。
    • CORS 仅允许可信来源。
    • 对敏感修改接口额外校验来源与用户权限。

低危（记录、长期改进）

E. IDOR（水平/垂直越权）在前端 API 参数层面

  • 描述：部分资源通过可控 ID 暴露，若后端未做完整授权校验，攻击者可请求他人资源。例如：GET /user/profile/{id} 参数改为其他用户 id。
  • 影响：隐私泄露、数据不一致。
  • 复现步骤：
    1. 登录用户 A，访问 /api/user/123/profile。
    2. 将 123 替换为其他用户 id，观察是否返回数据。
  • 建议修复：
    • 服务端在接口侧强制校验用户权限，基于登录主体判断资源访问权限，禁止仅依赖前端判断。

F. 前端依赖库老旧导致组件级漏洞

  • 描述：若前端使用过期第三方库（富文本、文件处理、加密库等），可能含已知漏洞。
  • 影响：被利用导致 XSS、敏感数据泄露或加密机制失效。
  • 建议修复：
    • 进行依赖清单扫描，升级到无已知漏洞的版本。
    • 对外部脚本来源增加完整性校验（SRI），或本地化依赖并纳入版本管理。

8 风险评级说明

  • 高危：容易被远程利用，可能导致重要资产泄露或系统接管。优先在 7 天内修复或采取临时缓解。
  • 中危：利用成本或可访问条件较高，但仍能造成严重影响。优先在 30 天内修复。
  • 低危：对业务影响有限，长期改进项。

第9节 — 修复建议（详细，可落地）

下面给出更为详细、可直接落地的修复建议，包含代码片段与实现要点，便于开发直接在项目中应用并回归验证。

9.1 不在前端或 Cookie 中保存明文敏感信息（会话与凭证处理）

要点：仅在服务器端生成短期会话凭证；前端仅保存非敏感标识（或把凭证保存为 HttpOnly, Secure cookie）。

Spring Boot（使用 Spring Security）示例：

实现要点：

  • 不在 cookie/localStorage 中存储明文密码或密钥。
  • 使用 HttpOnly + Secure + SameSite 限制 cookie 访问。
  • 若使用 JWT，后端使用对称/非对称密钥签名，前端仅保存 token，不保存私钥。
  • token 采用短期过期 + 刷新机制，刷新接口需做额外校验（设备、IP、频率）。

9.2 文件下载路径校验与白名单（防止路径遍历、任意文件读取）

要点：使用映射 ID（或数据库记录）而非直接传递文件系统路径；在处理路径时做正规化与基目录校验。

安全实现示例（Java）：

实现要点：

  • 上传文件写入数据库记录（fileId, originalName, storagePath, ownerId），对外仅使用 fileId 访问。
  • 对 name 参数做严格校验，拒绝包含 .. 或 URL 编码后的遍历序列。
  • 对关键目录设置只读、不可执行权限，避免 webserver 直接解析上传目录中的脚本文件。

9.3 XSS 防护（输入白名单 + 输出编码）

要点：前端与后端双重防护。前端对富文本使用可信清洗库（如 DOMPurify）。后端对所有用户可显示字段进行输出编码。

前端建议：

  • 在提交富文本前使用 DOMPurify 清洗：DOMPurify.sanitize(html, {ALLOWED_TAGS: [...]})。
  • 对 img 的 src 做域名白名单或代理转储为受控路径。

后端（Java）输出编码示例：

实现要点：

  • 富文本仅允许必要标签（a、p、img、strong、em 等），禁止 script、iframe 等。
  • 对所有在 HTML 上直接渲染的字段做 escape 转义。

9.4 CSRF 与 CORS（防止跨站请求与跨域滥用）

CSRF 建议（Spring Security）：

  • 对表单提交启用 CSRF Token。
  • 对 REST API，可使用双重提交 Cookie（Cookie + 自定义 header）或在 Authorization header 中传递 token 并校验。

CORS 建议：

  • 不要使用 Access-Control-Allow-Origin: *。
  • 在服务端列出允许的前端域名，启用 allowCredentials(true) 时严格匹配源。

实现要点：

  • 写操作必须验证 CSRF Token 或 Authorization header。
  • 检查 preflight 响应，仅在可信 origin 时返回允许跨域头。

9.5 授权校验（防止 IDOR）

要点：服务端基于认证主体校验资源访问权限，前端逻辑仅作用户体验控制。

示例：

实现要点：

  • 在控制层或服务层统一校验权限。
  • 对外接口返回的资源应掩码敏感字段，按最小权限原则展示数据。
  • 使用注解（@PreAuthorize）或 AOP 切面集中处理授权逻辑。

9.6 前端依赖与第三方库管理

要点：自动化依赖扫描与周期性升级。

实践建议：

  • 在 CI 中执行 npm audit、npm ci，对高危漏洞阻断发布。
  • 对后端使用 SCA 工具扫描 Maven/Gradle 依赖。

9.7 文件上传（防止上传 WebShell）

要点：文件类型检测、重命名、权限限制、隔离存储。

实现示例：

实现要点：

  • 使用 libmagic 或 Apache Tika 检测真实 MIME 类型。
  • 阻止危险扩展（.jsp .php .sh 等）。
  • 上传目录不放在 webroot 下，使用后端接口提供文件内容。

9.8 审计日志与告警

要点：记录关键事件，建立告警规则。

建议日志字段：时间、请求者 ID、操作类型、目标资源、来源 IP、User-Agent、结果。

实现要点：

  • 将敏感操作（登录失败超过阈值、文件下载大于指定次数）纳入告警策略。
  • 审计日志单独存储并保护，不应对外可写。

9.9 回归验证与自动化测试

要点：修复后增加自动化回归用例并纳入 CI。

实践建议：

  • 单元测试覆盖授权与异常分支。
  • 集成测试模拟路径遍历、IDOR、XSS 用例。
  • 在 CI 中加入 SAST 与依赖扫描任务，阻断高危问题。

第10节 — 后续工作建议

  • 在隔离环境对每一项高危问题进行可重复复现并生成 PoC。
  • 修复后在 QA 环境进行回归测试并记录结果。
  • 在 CI/CD 中加入静态安全扫描（SAST）、依赖安全扫描（SCA）与自动化回归测试。
  • 建议定期（季度）进行第三方依赖与镜像的安全评估。
