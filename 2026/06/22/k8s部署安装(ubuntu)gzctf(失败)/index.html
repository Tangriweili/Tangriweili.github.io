常见问题：

  • Postgres crashloop：kubectl -n gzctf logs deploy/postgres。

6) 部署 GZCTF 应用 Deployment + Service

保存为 gzctf-deploy.yaml 并 apply。

要点：

  • 镜像用 gztime/gzctf:latest（如需固定 tag 可改）

  • DATABASE_CONNECTION 直接从 Secret 中读取（我把完整连接串放在 Secret 的 DATABASE_CONNECTION 字段）

  • REDIS_CONNECTION 指向 redis:6379

  • GZCTF 监听容器内 8080，Service 暴露 80（Ingress 使用 80）

Apply：

排查：

  • 如果 Pod 一直 CrashLoopBackOff：kubectl -n gzctf logs deploy/gzctf 查看错误（通常是 DB 连接错误或缺少 XOR_KEY）。

  • 如果应用日志显示 DB migration 成功，则管理员密码在 Secret 中的 GZCTF_ADMIN_PASSWORD 生效。

9) 验证访问流程（按顺序检查）

  1. 查看所有资源是否就绪：
  1. 检查 GZCTF Pod 日志，等待迁移完成：

  • 迁移成功后日志会提示 DB 表创建或 migration 完成。

  • 如果日志出现 Unable to connect to database，检查 kubectl -n gzctf get svc，确保 postgres service 存在，并尝试从 busybox / alpine pod 内连通测试：

  1. 访问页面：
    在浏览器里打开 http://ctf.com/ 应该能看到 GZCTF 页面（首次启动如果 DB 初始化成功，会自动创建管理员，使用 secret 中的 GZCTF_ADMIN_PASSWORD 登录，如果应用文档说明不同，请以应用日志为准）。

  2. 如果 Ingress 无响应：

  • 检查 ingress-nginx 控制器是否运行：kubectl -n ingress-nginx get pods

  • 检查 Ingress 资源：kubectl -n gzctf describe ingress gzctf-ingress，查看 endpoints/backends 是否已填。

11) 可选加强（建议在实验稳定后考虑）

  • 把 Postgres 改成 StatefulSet + PVC（每个节点都有独立 PV），或使用外部托管 DB。

  • 在 Ingress 上启用 TLS（cert-manager + Let’s Encrypt），或在内网用自签证书。

  • 把 Secret 用外部 Vault 管理。

  • 把 hostPath PV 换成 local-path-provisioner，实现更灵活的本地存储。
